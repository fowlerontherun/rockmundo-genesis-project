# Event Framework Technical Design Plan

## Objective

Plan a future event framework that lets existing Rockmundo systems emit durable, queryable domain events without changing application code in this planning PR. The framework should start as an append-only audit and integration layer, then mature into a safe foundation for notifications, analytics, admin tooling, achievements, automation, and player-facing timelines.

## Current System Touchpoints That Could Emit Events Later

The codebase already has several systems that create outcomes worth capturing as formal events:

- **Random and admin-triggered player events**: event assignment, player choice, and delayed outcome completion from `random_events` and `player_events` flows.
- **Prison events**: prison choice resolution and release outcomes.
- **Career milestones**: milestone tracking, journal entries, morale boosts, and career progression moments.
- **Gig, open mic, tour, and major event performances**: booking, start, song scoring, completion, payouts, fan changes, and crowd responses.
- **Music production and releases**: songwriting, recording, video production, manufacturing, streaming, radio, charts, and release lifecycle changes.
- **Economy and monetisation**: purchases, checkout fulfillment, VIP or slot flows, donations, sponsorship payments, weekly band pay, and merch sales.
- **Social and media systems**: Twaater activity, bot engagement, PR activities, radio invitations, and demo reviews.
- **Health, morale, reputation, wellness, work, travel, education, family, politics, and housing systems**: scheduled activity completion, band health changes, university attendance, family daily ticks, elections, mayor salaries, housing market updates, and related state transitions.

The future framework should not replace these systems initially. It should let them publish normalized event records alongside their existing writes.

## Design Principles

1. **Append-only by default**: events describe facts that happened and should not be edited except for explicit redaction or repair workflows.
2. **Domain-first naming**: event names should describe game concepts, not implementation details.
3. **Low coupling**: emitters should not need to know every downstream consumer.
4. **Idempotent writes**: every emitter should provide a deterministic idempotency key where practical.
5. **Privacy-aware payloads**: avoid storing secrets, payment provider raw payloads, prompts, or personally sensitive data unless explicitly required.
6. **Incremental adoption**: begin with a database-backed outbox/event log and only add queues or workers once clear scale needs appear.
7. **Observable failures**: event write failures should be visible, retryable, and should not silently corrupt game state.

## Event Shape

Recommended canonical event envelope:

```ts
type DomainEvent = {
  id: string;                    // UUID generated by the event store
  event_name: string;            // e.g. "gig.completed"
  event_version: number;         // starts at 1; increments for breaking payload changes
  category: EventCategory;       // broad routing and reporting group
  occurred_at: string;           // ISO timestamp for when the fact happened
  recorded_at: string;           // ISO timestamp for when the event was persisted
  producer: string;              // function, client module, cron job, or admin tool
  environment: string;           // local, preview, staging, production

  actor: {
    type: "profile" | "band" | "admin" | "system" | "npc";
    id: string | null;
  };

  subject: {
    type: string;                 // e.g. "gig", "song", "release", "player_event"
    id: string | null;
  };

  related_entities?: Array<{
    type: string;
    id: string;
    role?: string;                // e.g. "band", "venue", "city", "song"
  }>;

  idempotency_key?: string;       // unique when event can be retried
  correlation_id?: string;        // groups events from one request/job/run
  causation_id?: string;          // parent event id when one event caused another

  visibility: "internal" | "player" | "band" | "public";
  severity?: "debug" | "info" | "warning" | "critical";

  payload: Record<string, unknown>;  // typed per event_name + event_version
  metadata?: {
    request_id?: string;
    job_name?: string;
    source_table?: string;
    source_record_id?: string;
    schema_version?: number;
  };
};
```

### Naming Conventions

- Use lowercase dot-separated past-tense names: `gig.completed`, `song.recorded`, `random_event.choice_submitted`.
- Reserve present-tense commands such as `gig.complete_requested` only when capturing a request rather than a completed fact.
- Keep event versions independent from application release versions.
- Prefer stable IDs over names in payloads; duplicate names only when they improve audit readability.

## Event Categories

| Category | Purpose | Example event names |
| --- | --- | --- |
| `gameplay` | Player-facing game actions and outcomes | `random_event.assigned`, `random_event.outcome_applied`, `prison_event.choice_resolved` |
| `career` | Artist, song, fame, chart, milestone, and release progression | `career.milestone_unlocked`, `song.audio_generated`, `release.manufacturing_completed` |
| `performance` | Gigs, open mics, major events, tours, rehearsals, jam sessions | `gig.started`, `gig.completed`, `major_event.song_scored`, `tour.arrived` |
| `economy` | Currency, payouts, purchases, sponsorships, donations, merch | `band.weekly_pay_processed`, `checkout.fulfilled`, `sponsorship.payment_processed` |
| `social` | Twaater, PR, radio, media, bot and audience interactions | `twaater.post_boosted`, `radio.invitation_generated`, `pr_activity.completed` |
| `simulation` | Scheduled ticks, automated jobs, NPC actions, market updates | `daily_sales.generated`, `housing_market.updated`, `npc_mayor.vote_cast` |
| `health` | Morale, reputation, wellness, health decay and recovery | `band_health.morale_changed`, `wellness.activity_completed` |
| `admin` | Admin interventions, manual repairs, moderation | `admin.random_event_triggered`, `admin.plays_boosted` |
| `system` | Infrastructure, job lifecycle, retries, errors | `job.started`, `job.completed`, `event_publish.failed` |

## Database Implications

### Proposed Tables

1. **`domain_events`**
   - Stores canonical append-only events.
   - Recommended columns:
     - `id uuid primary key default gen_random_uuid()`
     - `event_name text not null`
     - `event_version integer not null default 1`
     - `category text not null`
     - `occurred_at timestamptz not null`
     - `recorded_at timestamptz not null default now()`
     - `producer text not null`
     - `actor_type text not null`
     - `actor_id uuid null`
     - `subject_type text not null`
     - `subject_id uuid null`
     - `visibility text not null default 'internal'`
     - `severity text null`
     - `idempotency_key text null unique`
     - `correlation_id uuid null`
     - `causation_id uuid null references domain_events(id)`
     - `payload jsonb not null default '{}'::jsonb`
     - `metadata jsonb not null default '{}'::jsonb`

2. **`domain_event_entities`**
   - Optional normalized index of related entities for efficient lookup.
   - Columns: `event_id`, `entity_type`, `entity_id`, `role`.
   - Add indexes on `(entity_type, entity_id, recorded_at desc)` and `(event_id)`.

3. **`domain_event_delivery_attempts`**
   - Optional table for later fan-out to notification, analytics, webhook, or job consumers.
   - Columns: `event_id`, `consumer_name`, `status`, `attempt_count`, `last_error`, `next_retry_at`, `delivered_at`.

4. **`domain_event_subscriptions`**
   - Optional future admin-configured subscription registry.
   - Columns: `consumer_name`, `event_name_pattern`, `category`, `enabled`, `config`.

### Indexing and Retention

- Add btree indexes for `event_name`, `category`, `actor_type/actor_id`, `subject_type/subject_id`, `recorded_at desc`, and `correlation_id`.
- Add GIN index on `payload` only after query patterns justify the write overhead.
- Consider monthly partitioning if the table grows past operational comfort or high-volume simulation ticks become event-backed.
- Define retention by visibility and category:
  - Player-visible career and gameplay events: long-lived.
  - Internal debug/system events: shorter retention or archive.
  - Payment and admin events: retain according to compliance and audit needs.

### Row-Level Security

- Service-role emitters can insert all events.
- Players should only read events with `visibility in ('player', 'band', 'public')` and matching actor, subject, or related entity membership.
- Admin reads should remain behind existing admin authorization.
- Do not expose internal payload fields unless a dedicated read model strips sensitive metadata.

### Transaction Boundaries

- Preferred initial approach: write domain events in the same database transaction as the state change when the code path already owns a database transaction.
- Supabase Edge Functions that perform multiple independent writes should use an idempotent event insert after the source-of-truth write succeeds.
- If event writing fails, default behavior should be configurable per event:
  - Critical audit events fail the operation.
  - Non-critical analytics events log and retry through a dead-letter mechanism.

## Migration Plan

### Phase 0: Documentation and Alignment

- Approve the event envelope, naming conventions, and category list.
- Identify the first systems that benefit from event capture without changing gameplay behavior.
- Decide which events are audit-critical versus analytics-only.

### Phase 1: Schema Foundation

- Add `domain_events` and `domain_event_entities` migrations.
- Add database constraints for category, visibility, event version, and idempotency.
- Add RLS policies for service-role inserts and carefully scoped reads.
- Create a small database helper or Edge Function shared helper for publishing events.

### Phase 2: Low-Risk Emitters

- Add emitters to flows that already create log-like records, such as career milestones, random event outcomes, prison event choices, and completed gigs.
- Keep existing `activity_feed`, journal, milestone, and health event writes unchanged.
- Add tests for idempotency keys and event payload shape.

### Phase 3: Read Models and Admin Visibility

- Create internal admin views for event search by player, band, subject, and correlation ID.
- Build derived read models only where needed, such as player timelines or band audit history.
- Avoid replacing current player-facing feeds until event coverage is consistent.

### Phase 4: Fan-Out Consumers

- Introduce subscription and delivery attempt tables or a queue-backed equivalent.
- Add consumers for notifications, analytics, achievements, or scheduled follow-up actions.
- Ensure consumers are idempotent and can be replayed from `domain_events`.

### Phase 5: Backfill and Consolidation

- Backfill selected historic milestones, gigs, releases, and random event outcomes using conservative payloads.
- Mark backfilled events with metadata such as `{ "backfilled": true, "source_table": "..." }`.
- Evaluate whether older feed/log tables should remain source-specific or be gradually converted into read models.

## Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Event payloads become inconsistent | Consumers break or analytics become unreliable | Maintain a typed event registry and validate payloads before insert |
| Duplicate events from retries | Inflated notifications, rewards, or metrics | Require idempotency keys for retried functions and unique constraints where possible |
| Sensitive data leaks through payloads | Privacy and compliance issues | Define blocked fields, use allowlisted payload builders, and expose read models instead of raw events |
| Event writes slow gameplay paths | Player-visible latency increases | Keep inserts minimal, defer non-critical fan-out, and add indexes carefully |
| Events are mistaken for source of truth too early | Hard-to-debug data conflicts | Treat events as audit/integration records until specific projections are designed |
| High-volume jobs create excessive storage | Database bloat and slower queries | Categorize noisy events, apply retention, and use partitioning or sampling for system diagnostics |
| Schema evolution is unmanaged | Old consumers fail after payload changes | Version every event shape and support multiple versions during migrations |
| RLS policies are too broad | Players see events they should not see | Test policies with representative profile, band, admin, and anonymous contexts |

## First 5 Implementation PRs

1. **PR 1: Event schema foundation**
   - Add migrations for `domain_events` and `domain_event_entities`.
   - Add constraints, indexes, and initial RLS policies.
   - Include database tests for insert permissions, uniqueness, and read restrictions.

2. **PR 2: Shared event publisher utility**
   - Add a shared Edge Function helper for building and inserting canonical events.
   - Include event name constants, category constants, and basic payload validation.
   - Add unit tests covering required envelope fields and idempotency behavior.

3. **PR 3: Gameplay audit emitters**
   - Emit `random_event.assigned`, `random_event.choice_submitted`, `random_event.outcome_applied`, and `prison_event.choice_resolved`.
   - Preserve all current tables and player-facing behavior.
   - Add tests for duplicate-safe retries.

4. **PR 4: Career and performance emitters**
   - Emit `career.milestone_unlocked`, `gig.completed`, `major_event.song_scored`, and `tour.arrived`.
   - Confirm event writes do not alter reward calculations or activity feed output.
   - Add integration tests around representative completion flows.

5. **PR 5: Admin event viewer and operational tooling**
   - Add an admin-only event search view or page backed by RLS-safe queries.
   - Include filters for player, band, event name, category, correlation ID, and time range.
   - Add a dead-letter or failed publish inspection path if publisher retries are introduced.

## Key Recommendations

- Start with a **database-backed append-only event log**, not a full queue or event-sourcing rewrite.
- Keep current source-of-truth tables and gameplay logic unchanged during the first adoption phases.
- Require **idempotency keys** for Edge Functions, cron jobs, webhook handlers, and admin actions that may retry.
- Create a **typed event registry** before adding many emitters so naming, payload versions, and validation do not drift.
- Use events first for **auditability and observability**, then add fan-out consumers once the core event log is proven stable.
- Expose player-facing timelines through **read models**, not raw `domain_events`, to avoid leaking internal metadata.
