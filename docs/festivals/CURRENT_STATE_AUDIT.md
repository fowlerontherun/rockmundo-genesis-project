# Festival domain current-state audit

## Executive summary

Repository inspection confirms **two overlapping festival systems plus several bridge/partial feature layers**:

1. **Legacy event-backed player flow**: `game_events` rows with `event_type = 'festival'`, `festival_participants`, and `src/hooks/useFestivals.ts`. This powers the registered `/festivals` browser and direct performance settlement.
2. **Dedicated festival ownership/domain flow**: `festivals`, festival ownership/sale/staff/permit/insurance/expense-ledger tables, and owner/admin pages such as `FestivalOwnerConsole`, `FestivalRunWizard`, `FestivalDirectory`, and `CityFestivalsAdmin`. This appears newer because its migrations are dated `20260714` and `20270420`, after the original `game_events` and `festival_participants` migrations.
3. **Hybrid expansion tables**: `festival_stages`, `festival_stage_slots`, tickets, attendance, watch rewards, finances, quality ratings, slot offers, negotiations, backstage events, performance history, reviews, merch, sponsorships and rivalries mostly still reference `game_events` rather than `festivals`.

The main risks are split authority, duplicated admin/player pages, status strings with incompatible meanings, direct client-side money/fame settlement, schedule entries created separately from booking authority, and UI-only features such as setlists that do not yet have persistent festival-specific storage.

Recommended canonical direction: **dedicated festival tables become authoritative**. `festivals` should represent the permanent brand/organisation, a later `festival_editions` table should represent dated occurrences, and stage, slot, application, contract, setlist, performance and operational tables should hang off editions. `game_events` may reference a festival edition for calendar/world presentation but must not remain authoritative. Legacy `festival_participants` should be migrated into canonical applications/contracts/slots/performances. Financial and reward settlement should become server-authoritative and idempotent.

### Views, materialized views, enums, indexes, triggers, policies

No festival-specific views or materialized views were found in generated Supabase types. Festival-related enums are not dedicated festival enums; `show_type_enum` includes `festival` and city law fields include `festival_permit_required`. The migrations create many indexes named `idx_festival_*`, `festivals_*_idx`, `festival_lineups_*_idx`, and policies that are often broad (`Authenticated users manage stages/slots/finances`) in early hybrid migrations. Triggers found include `trg_festivals_updated_at` on `festivals` and `trg_notify_festival_slot_offer` on `festival_slot_offers`.

## Edge Functions and scheduled activity integration

No Edge Function directory is named for festivals. Repository search did not find a dedicated festival Edge Function. Scheduled integration exists in `src/hooks/useFestivals.ts`: `applyToFestival` calls `createScheduledActivity` with activity type `other`, festival metadata, and dates from `game_events`. Schedule conflict checks in `src/hooks/useFestivalScheduleConflict.ts` exclude `completed` and `cancelled` activities and compare date windows.

## Implementation progress — canonical editions foundation

The canonical edition foundation PR introduces `festival_editions` as the dated occurrence model beneath permanent `festivals` brand rows. It adds a server-validated edition lifecycle RPC, immutable lifecycle history, dedicated-festival backfill into first editions, and explicit legacy mappings for dedicated festival rows plus conservatively matched festival `game_events`.

Legacy event/player flows remain active during this migration. Stage, slot, ticket, attendance, application, contract, setlist, performance and settlement tables are not re-keyed in this step and remain outstanding for later PRs.

## Canonical edition hardening update

The corrective edition hardening PR retains the historical `20291204090000_create_festival_editions.sql` migration because this checkout cannot prove it has not been applied in a shared Supabase environment. A follow-up additive migration, `20291205090000_harden_festival_editions.sql`, fixes the public-view dependency ordering, recreates the public-safe view, adds creation/transition idempotency request records, changes planning updates to JSONB patch semantics, and strengthens lifecycle validation.

Owner workflow edits to attendance, ticket range, dates and title now target canonical editions when present, while permanent festival brand edits remain separate. ## Canonical booking/contracts addition

The next canonical layer introduces edition-scoped applications, offers, immutable revisions, signed contracts, setlists and private audit events.
